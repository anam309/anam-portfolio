Case File RAP-2026 · Data Engineering

Retail Analytics Platform

A Python + PostgreSQL pipeline that moves raw retail extracts into a dimensional warehouse, with SCD Type 2 history, an idempotent load path, and a data-quality framework that keeps a paper trail on every row it touches.

data/raw/*.csv → extract → transform → load → warehouse

01 · OVERVIEW

What's on the manifest

Three source files come in as a batch: customers, products, sales. Extract reads them with zero type coercion, so a malformed value fails loudly in transform rather than silently as a NaN. Transform validates, coerces, hashes, and resolves foreign keys. Load lands the batch in staging for audit, then upserts it into a proper star schema, one dimension version at a time.

02 · ROUTE

How a batch moves through

Dimensions load before facts are FK-resolved, deliberately. A customer's first purchase in the same batch as their first registration still has to find a real customer_sk, not fall back to the "unknown" sentinel.

STAGE 1

Extract

CSV → DataFrame, every column read as str. No type inference, no silent NaNs.

STAGE 2

Transform

Null checks, type coercion, range & business-rule checks, SCD2 hashing.

STAGE 3

Load dims

SCD2 upsert for dim_customer / dim_product; hash-unchanged rows are skipped.

STAGE 4

Load facts

FK lookups now resolve against this batch's own dims; bulk insert with dedup pre-filter.

03 · WAREHOUSE

Star schema

Two SCD Type 2 dimensions carry full history on every change. One static date dimension auto-extends when a transaction lands outside its pre-populated range. Every run writes its own receipt.

Dimension · SCD2
dim_customer

Natural key customer_id, surrogate customer_sk. Full history on name / contact / address / loyalty-tier changes.

Dimension · SCD2
dim_product

Natural key product_id, surrogate product_sk. Full history on name / category / price / cost changes.

Dimension · static
dim_date

Pre-populated 2020-2030. Auto-extends ±30 days around any date outside that range.

Fact
fact_sales

One row per (transaction_id, line_number). Pre-computed revenue, cost, and margin at load time: this month's cost changes don't retroactively rewrite last month's numbers. Deduped on _stg_sale_id before every insert.

04 · REPORTING

From warehouse to business report

Three materialized views turn the star schema into business-ready reporting. Refreshed automatically at the end of every successful run, so they're never staler than the last load.

Materialized view
mv_product_performance

Quantity sold, revenue, profit, and margin % per product.

Materialized view
mv_customer_ltv

Order count, total spent, average order value, and tenure per customer.

Materialized view
mv_channel_payment_analysis

Revenue and margin by channel and payment method, by month.

CLI
reports/generate.py

One command, python -m reports.generate, writes five CSVs plus an HTML summary with interactive SVG charts (revenue trend, top products, top customers, channel performance) alongside every table. The same views are exactly what a BI tool like Power BI or Looker would connect to directly.

05 · INSPECTION

Data quality framework

Nothing is silently dropped. Every failed row is classified and written to staging.stg_rejections with the full source record as JSON.

NULL_VIOLATION

Required field is blank or missing.

TYPE_ERROR

Field can't be cast to its target type.

RANGE_CHECK

Value outside a valid range: zero quantity, discount over gross.

BUSINESS_RULE

Logically impossible: a transaction dated tomorrow.

DUPLICATE

Row already exists in the warehouse.

REF_INTEGRITY

FK lookup returned no match (sales default to −1 instead).

06 · INCIDENT LOG

Five bugs, found by actually running it

A pipeline that only exists as read code is unreviewed. These surfaced by generating data and running the full load against a real Postgres instance. Each now has a regression test.

Resolved
Case 1 · etl/load.py

SCD2 upsert inserted a column that doesn't exist

extract.py tags every row with _source_file for provenance. The generic _scd2_upsert() passed that straight through into dim_customer / dim_product, which have no such column. Every dimension load failed before anything committed.

FIX: filter the insert to the destination table's actual columns via sqlalchemy.inspect.

Resolved
Case 2 · retail_analytics_schema.sql

The "unknown" sentinel had nowhere to point

fact_sales.customer_sk / product_sk default to −1 when a sale references a customer or product not yet on file. The foreign key constraints require −1 to exist as a real row. Nothing had ever seeded it, so every unresolved sale threw ForeignKeyViolation.

FIX: seed an UNKNOWN sentinel row in both dimensions.

Resolved
Case 3 · etl/load.py

Sales were FK-resolved before their own dimensions loaded

On a first-ever batch, a customer's first purchase and their first registration arrive together. The pipeline used to resolve sale FKs before loading that batch's dimension rows, so every same-batch customer and product looked unknown, 100% of the time.

FIX: reordered run_pipeline to load dimensions first, then resolve sales against them.

Resolved
Case 4 · generate_data.py + etl/*.py

Unicode arrows crashed the run (Windows only)

Log and print statements used → and box-drawing characters. Windows consoles default to the cp1252 codepage, which can't encode them. The script died mid-run with UnicodeEncodeError, after partially writing output.

FIX: plain ASCII in every log/print path.

Resolved
Case 5 · retail_analytics_schema_v2_improvements.sql

Materialized view refresh failed the first time anything called it

REFRESH MATERIALIZED VIEW CONCURRENTLY requires a unique index on the view. mv_channel_payment_analysis never had one. Nothing had ever actually called sp_refresh_materialized_views() until the reporting layer above was wired into the pipeline, which is exactly what surfaced it.

FIX: added a unique index on (channel, payment_method, month).
